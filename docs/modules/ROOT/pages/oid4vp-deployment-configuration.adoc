[[oid4vp-deployment-configuration]]
== OpenID4VP deployment configuration reference

The following `sd-jwt-authenticator` settings control how the verifier builds
OpenID4VP authorization requests and validates responses.

[cols="1,1,2"]
|===
|Setting |Example |Purpose

|`clientIdScheme`
|`x509_hash`
|How `client_id` is built in authorization requests. `x509_hash` computes the hash from the configured access certificate.

|`queryLanguage`
|`dcql_query`
|Request format for requested credential data (`dcql_query` or `presentation_definition`).

|`responseMode`
|`direct_post.jwt`
|Enables encrypted authorization responses (`response` form field carrying JWE).

|`customUrlScheme`
|`openid4vp://`
|Wallet invocation URL scheme embedded in authorization request links.

|`accessCertificate`
|`MIIC...`
|Base64 DER X.509 certificate injected into request object `x5c` header.

|`registrationCertificate`
|`eyJ...`
|Opaque registration certificate value exposed via `verifier_info`.
|===

Specification-driven notes:

- Per OpenID4VP Final 1.0, with `responseMode=direct_post.jwt`, wallets return the encrypted `response` parameter.
- The plugin advertises an ephemeral ECDH encryption key via `client_metadata.jwks`.
- If `accessCertificate` is omitted, the plugin falls back to a self-signed certificate strategy.

Generic example authenticator configuration (realm JSON import format):

[source,json]
----
{
  "config": {
    "clientIdScheme": "x509_hash",
    "queryLanguage": "dcql_query",
    "responseMode": "direct_post.jwt",
    "customUrlScheme": "openid4vp://",
    "accessCertificate": "<base64-der-x509-cert>",
    "registrationCertificate": "<registration-certificate-jwt-or-token>"
  }
}
----

If you provision Keycloak using Terraform or similar tooling, this typically maps to variables like:
`sdjwt_client_id_scheme`, `sdjwt_query_language`, `sdjwt_response_mode`,
`sdjwt_custom_url_scheme`, `sdjwt_access_certificate`, `sdjwt_registration_certificate`.

=== SPRIN-D wallet sandbox profile (example)

For the German wallet ecosystem profile (HAIP/OpenID4VP PID presentation), use the
generic options above with profile-specific values and constraints from:

- https://bmi.usercontent.opencode.de/eudi-wallet/wallet-development-documentation-public/latest/architecture-concept/flows/22-pid-presentation/[PID presentation flow]
- https://bmi.usercontent.opencode.de/eudi-wallet/developer-guide/rp/PID_Presentation/[Relying Party PID presentation guide]

In that profile, it is typical to use:

- `responseMode=direct_post.jwt`
- request-object signature and certificate material (`accessCertificate`, `registrationCertificate`)
- a custom wallet invocation URL scheme
